# Ghola v0.5 — On-device email generation

## North star

User taps **Email** tile, finished draft is on screen in ≤500ms. Draft reads like
the user wrote it, not like an AI. **Nothing leaves the device.**

The unlock: move compute from foreground (user-watching) to background
(user-not-watching), plus a personal LoRA fine-tune that bakes the user's voice
into the model weights.

## Phases

### Phase 0 — Gmail OAuth on-device
Today the Gmail OAuth tokens are stored in `ghola-cloud`'s `connected_accounts`
table; every Gmail call hops through the server. Switch to **AppAuth-Android**;
tokens land in `SecureStorage` (encrypted SharedPreferences, already in place).
Server route `/api/gmail/connect` becomes a no-op stub kept only for backwards
compat. `GmailClient.kt` becomes the on-device Gmail API wrapper with auto-refresh.

### Phase 1 — Gmail mirror + corpus indexing
`GmailMirrorWorker` (CoroutineWorker via WorkManager, constraints
`requiresCharging + requiresBatteryNotLow`) mirrors the user's sent folder into
a local SQLite table on first run, incremental nightly. Each row gets a
**384-dim float embedding** (BLOB column) from MiniLM-L6-v2 INT8 via ONNX
Runtime Android. Corpus never leaves the device.

### Phase 2 — Voice transfer via retrieval
`VoiceCorpus.findStyleAnchors(intent, recipient, k=5)` — pure Kotlin
brute-force cosine sim over the embedding column (1k rows × 384 dims < 50ms on
the Seeker CPU). Boost: same-recipient +0.2, same-thread-family +0.1.
`EmailPromptBuilder` assembles `anchors + recipient context + intent + rubric`
into a tight prompt with no AI tells.

### Phase 3 — Local LLM + skeleton-first streaming
**Phi-3 Mini Q4 (.task, ~2GB)** via MediaPipe `LlmInference`. Two-pass prompt:
pass 1 emits `{to, subject}` JSON in <300ms; pass 2 streams the body. UI
renders skeleton immediately, then appends body tokens via
`MutableStateFlow<String>`. Cold-tap budget: 3-5s total on Seeker, but
**perceived** latency is the first paint at ~400ms.

### Phase 4 — Background pre-drafting
FCM push receiver (or 5-min Gmail poll fallback). On new inbound email in a
thread the user has previously replied to, schedule `PreDraftWorker`. Draft is
generated in the background, cached in `EncryptedSharedPreferences` keyed by
`predraft:$thread_id` with 24h TTL. When user opens Email tile for that
thread, **draft is already there** — zero foreground latency.

### Phase 5 — Cursor-style autocomplete
`SuggestionEngine.kt` watches the body `EditText`, debounces 150ms after
typing stops, calls `LocalEmailService.complete(prefix, maxNewTokens=20)`.
`GhostTextOverlay` renders the suggestion as faded inline text below the
cursor. Tab/space accepts; any other input rejects.

### Phase 6 — Personal LoRA fine-tune (THE MOAT)
LoRA rank 16, 3 epochs, on the user's last 500 sent emails (intent ⇄ email
pairs derived by reverse-summarization). Runs overnight on charging + Wi-Fi
via `PersonalFineTuneWorker`. Output: ~50MB adapter, applied at inference
time. The base model stays generic; the adapter contains the user's voice.
Requires `llama.cpp` finetune binary cross-compiled to Android NDK
(`libllama_finetune.so`).

### Phase 7 — Speculative decoding + NPU compilation
TinyLlama 350M Q4 proposes 4-token chunks; Phi-3 Mini verifies in parallel.
Standard speculative decoding, 2-3x throughput. NPU compilation via MediaTek
Neuron SDK for Dimensity 9300 (Seeker), ~3-5x additional speedup. Combined
with the pre-drafting cache from P4, foreground generation becomes
effectively instantaneous.

## What ships in v0.5 (this push)

- **P0** — On-device Gmail OAuth.
- **P1** — Sent folder mirror + SQLite schema + ONNX MiniLM embedder.
- **P2** — `VoiceCorpus` retrieval + `EmailPromptBuilder`.
- **P3** — MediaPipe `LlmInference` integration with Phi-3 Mini Q4 +
  skeleton-first streaming.
- **P3b** — Email tile re-routed from `/api/emails/generate` (cloud) to
  `LocalEmailService` (device).

- **P7** — Speculative decoding + NPU compile (requires MediaTek SDK).

## Privacy audit

After v0.5 ships, the email path must satisfy:

1. With Wi-Fi + cellular **off** on the Seeker, the Email tile produces a
   draft (post initial model + corpus download).
2. `adb shell netstat` during a draft shows **zero** outbound connections to
   `*.ghola.xyz`, `*.onrender.com`, or `*.googleapis.com`. Only Send hits the
   network — and only Gmail's send endpoint, directly from the device.
3. The server-side `/api/emails/generate` route is **never** called from the
   Android client. (Server keeps the route only for web users without the
   on-device stack.)
4. The user's sent corpus + draft cache + LoRA adapter live exclusively in
   `EncryptedSharedPreferences` and the app's internal SQLite. None of it
   ever ships to a server, even for backup.
