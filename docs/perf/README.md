# Perf measurements

Phase-A baselines for the Apple-flash-memory adoption plan
(`.claude/plans/zesty-giggling-charm.md`).

## Files

- `baseline-local-llama-3.2-1b-q4f16.json` — captured measurements for
  the default Local-mode model.
- `sparsity-llama-3.2-1b.json` — generated by
  `scripts/measure-sparsity.py`; per-FFN-layer SwiGLU intermediate
  activation density.

## Capture procedure (one-time per device class)

1. **Cold load + first token.** Open `https://ghola.xyz/chat` in an
   incognito window on a freshly-cleared profile (so neither the
   service worker nor IndexedDB is warm). Open DevTools → Performance.
   Trigger a chat. After the first token streams in, paste into
   DevTools console:

   ```js
   const { snapshot } = await import("/_next/static/chunks/...perf-marks.js");
   copy(JSON.stringify(snapshot(), null, 2));
   ```

   Or, simpler — use the `performance.getEntriesByName("ghola:…")`
   pattern; the named marks are listed in `apps/web/src/lib/perf-marks.ts::MARKS`.

2. **Cache occupancy.** Same DevTools console:

   ```js
   const { snapshotCacheBytes } = await import("/_next/static/chunks/...sparsity-instrument.js");
   copy(JSON.stringify(await snapshotCacheBytes(), null, 2));
   ```

   Paste into the `measurements.cache_bytes_loaded` field of the
   baseline JSON.

3. **Peak VRAM.** Chrome only — `chrome://gpu` → "Memory" panel during
   the load. Approximate; the goal is rough order-of-magnitude.

4. **Tokens/sec warm.** After the model is loaded, send 3 medium-length
   prompts back-to-back. Compute (output_tokens / wall_time).

## Sparsity sidecar

Runs out of band (no WebGPU, no MLC — straight transformers):

```bash
cd /Users/andersonobrien/Downloads/ghola
python -m venv .venv && source .venv/bin/activate
pip install transformers torch
python scripts/measure-sparsity.py \
  --model meta-llama/Llama-3.2-1B-Instruct \
  --output docs/perf/sparsity-llama-3.2-1b.json
```

The script reports `overall_mean_active_fraction` and a `decision`
field. Per the plan:

- `<= 0.50` → Phase C (sparsity prediction + sliding window) is
  worthwhile. Expected ~30-40% latency win.
- `0.51 - 0.60` → marginal. Ship Phase B + E first; revisit C later.
- `> 0.60` → skip Phase C. Ship Phase B (bundling) + Phase E (smaller
  stronger models) only.
